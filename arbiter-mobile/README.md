# Arbiter Mobile

Expo / React Native client for the ARBITER mission-control dashboard. Pairs with
`arbiter-mission-control/` (FastAPI + LLM stack). The phone client gives Sir
Luke a voice-first command surface for the operator, with HUD-style data
screens, a Skia orb, and a streaming chat that renders the server's analysis
panels natively (with a Chart.js WebView fallback for chart-heavy sections).

## Layout

```
arbiter-mobile/
├── app/                      Expo Router screens (index = home, +modals)
├── components/
│   ├── Background/           TronBackground (Skia grid + radial bloom)
│   ├── Chat/                 ChatDrawer + reducer + tests
│   ├── HUD/                  Shared panel/stat/sparkline/status primitives
│   ├── Orb/                  Skia orb (simulation, frame builder, view)
│   ├── Voice/                useVoiceSession (expo-speech-recognition)
│   ├── panels/               PanelCard dispatcher + native panels + WebView fallback
│   └── Settings/             SettingsPanel (host URL + API key)
├── lib/
│   ├── api.ts                Auth-aware fetch wrapper, sendChat, streamChat
│   ├── sse.ts                Fetch-based Server-Sent Events parser (no EventSource)
│   ├── credentials.ts        Hooks: useApi, useCredentials
│   ├── storage.ts            SecureStore wrapper
│   └── types.ts              Wire shapes mirroring server.py
└── assets/                   Static assets (PanelWebView HTML is inlined in TS)
```

## Standard checks

```bash
cd arbiter-mobile
npx tsc --noEmit
npx jest
```

All tests run under `ts-jest` against `.ts` files only — pure logic lives in
`.ts` modules (`panelHelpers.ts`, `chatReducer.ts`, `orbSimulation.ts`,
`frameToDrawables.ts`, etc.) so the renderer JSX never has to be transformed
for tests.

## Connectivity — Tailscale Funnel

The operator's mission-control FastAPI server runs on the Mac M1 Pro. The
phone needs HTTPS to call it from cellular / external Wi-Fi. We use
[Tailscale Funnel](https://tailscale.com/kb/1223/funnel) to expose a single
public HTTPS endpoint, terminated at the operator's machine.

1. Install Tailscale on the Mac and join your tailnet.
2. Enable Funnel for the machine in the
   [admin console](https://login.tailscale.com/admin/acls) (ACL needs
   `"nodeAttrs":[{"target":["tag:operator"],"attr":["funnel"]}]` or
   equivalent).
3. From the Mac, while `arbiter-mission-control/server.py` is running on
   `:8000`:
   ```bash
   tailscale serve --https=443 --bg http://localhost:8000
   tailscale funnel 443 on
   tailscale funnel status     # confirms the public URL
   ```
4. Note the printed `https://<machine>.<tailnet>.ts.net` URL — that's the
   host URL the mobile app expects.
5. In the mobile app, open Settings → set **Host URL** to the funnel URL and
   **API key** to whatever you set in `ARBITER_API_KEY` in
   `arbiter-mission-control/.env`. Settings persist to SecureStore.

To take the funnel down: `tailscale funnel 443 off`.

> Cellular networks aggressively kill idle long-lived connections. The
> SSE chat stream (`/api/jarvis/chat/stream`) emits keepalive comments every
> ~25 s from the server side, but expect the occasional reconnect on flaky
> links. The non-streaming `/api/jarvis/chat` endpoint is unaffected.

## Builds — EAS dev client + internal distribution

The app uses several native modules that aren't in Expo Go
(`@shopify/react-native-skia`, `react-native-reanimated@4`,
`expo-speech-recognition`, `react-native-webview`). Use EAS to produce a
**dev client** binary you install on the device once and then push JS-only
updates via `npx expo start`.

```bash
# One-time: install + authenticate
npm install -g eas-cli
eas login

# One-time per project: link to your EAS project (creates eas.json)
eas init

# Build the dev client (iOS internal distribution / Android APK)
eas build --profile development --platform ios
eas build --profile development --platform android
```

Recommended `eas.json` profile shape (only relevant fields shown):

```jsonc
{
  "build": {
    "development": {
      "developmentClient": true,
      "distribution": "internal",
      "ios":     { "simulator": false },
      "android": { "buildType": "apk" }
    },
    "preview": { "distribution": "internal" },
    "production": { "autoIncrement": true }
  }
}
```

After the build finishes:

- **iOS** — install the `.ipa` via the EAS-provided link
  (TestFlight-style internal distribution). The device needs to be on the
  provisioning profile generated by EAS (`eas device:create` to enrol).
- **Android** — install the `.apk` directly from the link.

For day-to-day work, run `npx expo start --dev-client` on the Mac and scan
the QR code from the installed dev client — JS reloads, native modules stay
the way they were when the binary was built. Rebuild only when native deps
change (mic permission, new module, new entitlement).

## Permissions

`app.json` already declares:

- `NSMicrophoneUsageDescription`, `NSSpeechRecognitionUsageDescription`
  (iOS) — required for the push-to-talk orb tap.
- `android.permission.RECORD_AUDIO` — same on Android.

If you add a new native module that needs permissions, update `app.json`
and rebuild the dev client.

## Server-side requirements

The mobile client expects these endpoints on `arbiter-mission-control/server.py`:

| Path | Purpose |
|---|---|
| `GET  /api/auth/check` | Bearer-token validation |
| `POST /api/jarvis/chat` | Non-streaming chat (fallback) |
| `POST /api/jarvis/chat/stream` | SSE chat — used by `streamChat` |
| `GET  /api/revenue/summary` | Business Summary screen |
| `GET  /api/status` | Uptime screen |
| `GET  /api/gcp/summary` | Uptime screen (GCP regions) |
| `GET  /api/ceo/agents`, `/api/agents` | Orchestration screen |
| `POST /api/jarvis/vision` | Reserved for vision attachments |

Set `ARBITER_API_KEY` in the server `.env`; the mobile app sends it as
`Authorization: Bearer …` on every request.
